# Adversarial verification prompt

Use this prompt from the repository root to decide whether the goal in
[`GOAL.md`](GOAL.md) is actually complete.

## Role

You are an adversarial verifier. Assume the implementation is incomplete until
direct evidence proves otherwise. Inspect the repository and run the commands;
do not accept README claims, implementation summaries, cached prose reports, or
the presence of files as proof of behavior.

Do not edit source files while verifying. Temporary files must be created
outside the repository or under an already ignored verification-artifact
directory. Do not expose credentials or include environment-variable values in
the report.

## Required verdict format

Report every numbered condition as:

```text
R<number>: PASS|FAIL - <short evidence or exact failure>
```

Then report nice-to-have conditions as:

```text
N<number>: PASS|FAIL - <short evidence or exact failure>
```

End with exactly one of:

```text
VERDICT: PASS
VERDICT: FAIL
```

`VERDICT: PASS` is allowed only when every required condition passes. A command
that cannot run, a skipped live suite, a partial page or event stream, or missing
evidence is a failure, not an assumption.

## Required conditions

### R1 — Repository command contract

Inspect `package.json` and the referenced scripts. The repository must expose
these stable commands:

```sh
bun run check
bun run verify:offline
bun run verify:live
bun run verify
```

Run all four. Each must exit zero. `bun run verify` must run both offline and
live verification rather than aliasing only one side. Record exit codes and the
machine-readable report path emitted by the full verifier.

### R2 — Exact workflow mirror

Enumerate every regular `*.js` file directly under `.claude/workflows/` and
`.codex/workflows/` using `fd`; do not rely on a hard-coded expected count.

Required:

- The two directories have exactly the same basename set, including companion
  probes such as `parity-07b` and protocol suites such as `parity-12`.
- Every Codex file is derived from its same-named Claude fixture with only the
  intentional provider adaptations allowed by the repository's mirror checker.
- All `.claude/workflows` path references inside Codex fixtures point to
  `.codex/workflows`.
- Claude model `haiku` becomes `gpt-5.6-luna` everywhere in Codex fixtures.
- Claude model `sonnet` becomes `gpt-5.6-terra` everywhere in Codex fixtures.
- No active Codex fixture contains `haiku`, `sonnet`, `opus`, `fable`, or a
  `.claude/workflows` path.
- The Claude fixtures remain provider-original and are not rewritten to GPT
  model identifiers.

Run the repository's mirror/drift checker and include its compared-file total.

### R3 — App Server is the single live execution substrate

Inspect runtime imports, process spawning, and package dependencies.

Required:

- Live `agent()` execution uses `codex app-server` JSON-RPC over stdio.
- The runtime performs the required `initialize` request and `initialized`
  notification before other requests.
- Production workflow execution does not call one-shot `codex exec` and does
  not route through `@openai/codex-sdk`.
- The runtime does not depend on the experimental App Server WebSocket
  transport.
- Protocol messages are correlated by request ID and attributed by thread ID,
  turn ID, and item ID where supplied.
- Backpressure, malformed messages, process exit, and protocol errors fail
  explicitly rather than hanging indefinitely.

Start the real locally installed App Server through the runtime and prove a
successful initialize handshake. Record the Codex version without treating a
particular version number as permanently required.

### R4 — Current protocol and model discovery

Run `codex app-server generate-ts` into a temporary directory and ensure the
runtime's protocol use typechecks against the schema generated by the installed
Codex binary.

Through App Server, call the model-list surface and prove that the configured
models resolve to the literal IDs:

```text
gpt-5.6-luna
gpt-5.6-terra
```

Failure to discover either configured model is a failure before spending a live
workflow run.

### R5 — Script loading and sandbox boundary

Run offline tests proving that workflow files execute as plain JavaScript in an
async sandbox with the documented injected globals:

```text
agent parallel pipeline phase log workflow args budget
```

Required tests prove:

- top-level `await` and top-level workflow `return` work;
- `meta` is parsed without executing the workflow body;
- computed or malformed `meta` fails at load time;
- `require` and `process` are unavailable to workflow code;
- `Date.now()`, `Math.random()`, and argument-less `new Date()` throw;
- `new Date(ms)` and ordinary deterministic JavaScript built-ins work; and
- values crossing the sandbox boundary remain JSON-compatible and enforce the
  documented 4096-item cap.

### R6 — Deterministic orchestration semantics

Run offline tests, without GPT calls, proving:

- `parallel()` accepts thunks, preserves positions, waits for all slots, turns
  a throwing slot into `null`, and does not reject the whole barrier;
- `pipeline()` passes `(previousResult, originalItem, index)`, seeds stage one
  with the original item, streams each item independently through stages, and
  drops only the failed item while skipping its later stages;
- `phase()` and `log()` emit ordered workflow events;
- explicit per-agent phase attribution is not corrupted by concurrent work;
- `args` arrives verbatim, including `undefined`, objects, and strings;
- `budget.total`, `spent()`, and `remaining()` obey their documented contract;
  and
- an uncaught workflow error fails the run while documented fan-out failures
  remain visible as absorbed failures.

Include test counts, not only the test command's exit code.

### R7 — Composition, isolation, and lifecycle caps

Run offline and live evidence proving:

- `workflow()` resolves a child by `scriptPath`, forwards args verbatim, and
  returns the child's result;
- unknown workflows throw catchably;
- child-to-grandchild workflow nesting is rejected at the documented boundary;
- parent and child share the applicable agent, concurrency, cancellation, and
  budget accounting;
- `isolation: 'worktree'` gives the agent a distinct git toplevel, leaves the
  main checkout untouched, and cleans a worktree left clean;
- concurrency is bounded and excess agents queue rather than disappear; and
- lifetime and per-call caps fail visibly rather than truncating work.

### R8 — Authoritative text and structured results

Using real GPT calls through App Server, prove both modes:

- Without `schema`, `agent()` returns the completed agent's final text as a
  string.
- With `schema`, `agent()` returns an object conforming to the supplied JSON
  Schema, including nested objects, arrays, enums, integers, and item-count
  constraints.
- The runtime treats final `item/completed` state as authoritative rather than
  assuming concatenated deltas are the final result.
- A null, failed, interrupted, or schema-invalid result cannot be reported as a
  successful agent call.

The evidence must show at least one Luna call and one Terra call.

### R9 — Streaming progress is real and attributable

Run a live probe whose agent performs enough work to emit intermediate events.
Capture the normalized event stream from before the final result.

Required evidence includes:

- at least one agent message delta before `turn/completed`;
- at least one tool, command, file-change, plan, reasoning-summary, or
  collaboration lifecycle event before completion;
- explicit workflow run ID, agent ID, label, phase, model, thread ID, and turn
  ID attribution on normalized events where applicable;
- ordered `started`, intermediate, and `completed` lifecycle events; and
- token usage and terminal status attached to the completed agent.

A runner that buffers everything and only prints a final answer fails this
condition even if the final answer is correct.

### R10 — The orchestrator can interact with a running subagent

Run a live, controlled probe with a nonce generated by the verifier, not known
to the subagent's initial prompt.

Required sequence:

1. Start a subagent turn that remains active long enough to steer.
2. Receive attributable intermediate output from that subagent.
3. Send the nonce to the active turn through the runtime's orchestrator-facing
   messaging API, backed by App Server `turn/steer`.
4. Observe App Server accepting the steer for the expected active turn.
5. Receive subsequent attributable output or the final result containing the
   nonce.

Also run two sibling agents, interrupt one through the runtime, and prove the
other reaches successful completion. An implementation that can only send a
second prompt after the first turn finishes does not pass.

### R11 — Resume is exact-prefix memoization

Run the complete three-leg protocol documented for
`.codex/workflows/parity-12-resume.js`:

- R1: fresh run with salt `s1`; record run ID and nonces A, B, and C.
- R2: resume R1 with salt `s1`; require A, B, and C to be byte-identical to R1,
  with zero subagent tokens and a replay-scale duration.
- R3: resume R1 with salt `s2`; require A to equal R1, B to be fresh, and C to
  be fresh.

Inspect the journal evidence. Keys must be chained to prior execution history,
so a miss invalidates the later prefix even when a later call has the same
prompt and options. A per-`(prompt, opts)` cache that reuses R1's C in R3 fails.

Prove that the same completed run can be resumed more than once and that
replayed results remain byte-identical.

### R12 — Complete live parity sweep

Run every Codex workflow discovered in R2 through the real runtime. Do not stop
after the first failure and do not silently omit expensive or multi-leg suites.
Exercise all documented invocation modes, including:

- `parity-05` with omitted args, object args, and string args;
- `parity-07` composition and its nested companion;
- `parity-09` worktree isolation; and
- the full `parity-12` resume protocol from R11.

For each invocation, require the returned suite name to match, `passed` to be
`true`, and every non-INFO check to pass. Report:

- workflow files discovered;
- workflow invocations required;
- invocations completed;
- suites passed and failed;
- pending or skipped invocations;
- absorbed agent failures; and
- total live Luna and Terra calls.

`pending`, `skipped`, an unvisited discovered workflow, or an incomplete
multi-leg protocol makes this condition fail.

### R13 — Verification artifacts are reproducible and honest

The full verifier must write a machine-readable report and an append-only event
or transcript artifact for the current run. Inspect both.

Required:

- The report records command versions, start/end time supplied by the outer
  harness, discovered suite counts, per-condition results, per-suite results,
  model IDs, usage, failures, and artifact paths.
- Secrets, access tokens, authorization headers, and complete environment dumps
  are absent.
- Failed, skipped, interrupted, and pending work remain distinguishable.
- The full verifier exits nonzero whenever any required condition or suite
  fails.
- A second offline verifier run succeeds from a fresh process without relying
  on mutable in-memory state from the first.

### R14 — Negative controls can catch false success

Run the verifier's tests against temporary, deliberately broken fixtures. At
minimum prove it rejects:

- a missing Codex workflow mirror;
- a Codex fixture containing `model: 'haiku'`;
- a malformed or computed `meta` block;
- a suite result with `passed: false`;
- a structured result that violates its schema;
- an event stream with only a final response and no intermediate event; and
- a resume implementation that caches solely by `(prompt, opts)` and therefore
  incorrectly reuses C after B changes.

The real repository must not be modified to perform these negative controls.

### R15 — Morning review brief

Open the repository-root `BRIEF.html` in a browser and compare every factual
claim against the final machine-readable verification report and git history.

Required:

- The file is self-contained and readable without a development server or
  external network request.
- It explains the finished architecture and the boundary between the Bun
  workflow VM and Codex App Server.
- It reports the exact offline test totals, discovered workflows, required and
  completed live invocations, pass/fail/pending/skipped totals, Luna/Terra call
  totals, and final verifier verdict.
- It presents the streaming/steering/interruption proof and the three
  `parity-12` resume legs with links or paths to their evidence artifacts.
- It lists known limitations and nice-to-have failures honestly.
- It includes the logical implementation commit history for this goal.
- It contains no secret, access token, authorization header, complete
  environment dump, or unsupported success claim.
- The rendered page has no overlapping, clipped, or unreadable primary content
  at a normal desktop viewport.

Missing, stale, unrenderable, or factually inconsistent brief content fails
this condition.

## Nice-to-have conditions

### N1 — Rich collaboration visibility

The normalized event stream exposes App Server `collabToolCall` activity with
sender, receiver or new child thread, tool, status, prompt, and agent status
when Codex uses built-in collaboration tools.

### N2 — Human-readable live progress

In addition to JSON artifacts, the runner renders concise phase, agent, steer,
interrupt, failure, and completion updates suitable for an interactive terminal.

### N3 — Resume efficiency report

The verifier presents fresh-versus-resumed duration and token usage side by
side without turning environment-dependent timing into a brittle exact value.

## Final decision

Review all required results adversarially. If any evidence came only from a
mock where a live run was required, any discovered workflow or invocation was
not completed, any required event was inferred rather than captured, or any
command failed, print `VERDICT: FAIL`.

Print `VERDICT: PASS` only when R1 through R15 all pass.
